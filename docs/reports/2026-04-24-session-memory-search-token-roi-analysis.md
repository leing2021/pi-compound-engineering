# Session 记忆搜索 Token ROI 分析报告

> 日期：2026-04-24  
> 范围：本次 PU / Tri-Brain / Pi runtime / single_content / core-api 测试稳定化长会话  
> 目的：评估本 session 中“记忆搜索、上下文恢复、文档读取、工件检索”对 token 成本与产出价值的 ROI

---

## 一、结论摘要

### 总结论
本 session 的“记忆搜索 / 上下文恢复”整体 **ROI 为正，且在中后段非常高**，但前中段存在明显的 **过量上下文注入** 与 **低命中率预读**，导致 token 利用率不均衡。

可以概括为：

- **高 ROI 部分**：
  1. 压缩 summary 恢复长期状态
  2. 针对 worktree / plan / integration 证据的定点回溯
  3. 面向根因的最小文件集排查
  4. 把已解决问题沉淀为 solution / report / commit series

- **低 ROI 部分**：
  1. 一次性加载过多参考文件
  2. 对已基本确认无关的问题继续做宽范围读取
  3. 在“修复已确定”后仍重复读取大段历史上下文
  4. 将“记忆搜索”与“正式验证”混用，导致上下文冗余

### 本次 session 的总体判断
如果把 ROI 粗分为 5 档：

- **极高 ROI**：summary 恢复、worktree 状态回收、git diff/commit 分组、最小复现验证
- **高 ROI**：runtime-session 根因坐实、主仓合回、失败后最小回归定位
- **中 ROI**：技能规则与文档校验、solution 沉淀
- **低 ROI**：大批量 read-files 预载、与当前问题弱相关的参考文档一并读入
- **负 ROI 候选**：对同一问题多次宽范围扫描，但没有显著改变决策

整体上，这个 session **值得做记忆搜索**，因为任务跨度已经远超单轮短上下文能承载的复杂度；但搜索策略还可以更“分层化”和“按阶段限额”。

## 二、什么叫“记忆搜索 ROI”

这里的 ROI 不是精确 token 计费，而是工程视角的近似评估：

**ROI =（通过记忆检索减少的返工 + 减少的误判 + 减少的重复验证 + 增强的连续性） /（读取与注入这些记忆带来的 token 成本）**

本次主要看 4 类收益：

1. **连续性收益**
   - 长会话不丢状态
   - 能立即接着 worktree / test / integration 继续做

2. **决策收益**
   - 不重复争论已定路线
   - 直接沿批准的 P1/P2/P2-4 继续实施

3. **调试收益**
   - 快速回忆“先前已验证过什么”
   - 避免反复跑同样的错误假设

4. **交付收益**
   - 能把 worktree 中成果整理提交
   - 能在主仓回放并做差异定位

对应成本主要是：

- 读取历史 summary 的 token
- 读取文件、技能文档、方案文档的 token
- 把大量历史状态重新塞进当前窗口的 token
- 因上下文过大带来的注意力扩散和定位速度下降

## 三、本 session 中高 ROI 的记忆搜索模式

## 1. 压缩 summary 恢复是本场最高 ROI 动作之一

会话一开始就给出了高度结构化 summary，内容包括：

- 目标
- 约束
- 已完成进度
- 决策
- next steps
- 关键文件
- 已验证结果
- 未解决问题

这类 summary 的价值极高，因为它直接避免了重新做以下昂贵动作：

- 重新扫描 docs/tri_brain_system_brand_OS
- 重新审视 Pi 方案选择
- 重新梳理 P1/P2/P2-4 范围
- 重新确认 worktree 做过哪些改动
- 重新确认 integration smoke 的真实性边界

### ROI 判断
**极高 ROI**。

原因：
- 用一次压缩摘要替代了几十次 read / rg / bash / 比对动作
- 而且直接限定了“现在只该继续修 runtime-session.test.ts”
- 对长工程会话，这是典型的 10x 级别收益

### 可迁移经验
未来任何跨阶段工程任务，优先保留：
- 决策链
- 验证证据
- 未解决 blocker
- 关键文件路径

不要只保留“做了什么”，而要保留“为什么现在不该再重做某些事”。

---

## 2. 先用记忆缩小问题空间，再做最小验证，ROI 很高

在 `runtime-session.test.ts` 的续修里，真正高 ROI 的不是“读取了很多文件”，而是这条链路：

1. 先利用 summary 和记忆判断：
   - 问题大概率与测试隔离有关
   - 不应一开始就改业务代码
2. 再读少量关键文件确认：
   - `runtime-session.test.ts`
   - `app.ts`
   - `database.ts`
   - `client.ts`
3. 然后做最小实验区分：
   - 单跑通过？
   - 与 `tasks.test.ts` 合跑失败？
4. 最终把根因收敛到：
   - 固定 sandbox ticket id 命中唯一约束

这里“记忆”真正起作用的地方，是**提前告诉当前 agent：不要从零开始大面积排查**。

### ROI 判断
**高 ROI**。

因为：
- 记忆缩小了假设空间
- 验证动作非常便宜
- 最终产出了 test-only fix，而不是误改生产逻辑

---

## 3. 对已验证工件的回忆，比重新构造证据便宜得多

本 session 多次直接复用了既有事实：

- worktree 已经存在且可回滚
- `single_content` 已有 contract
- commit worker auto-finalize 已贯通
- HTTP smoke 已证明 `POST /tasks -> worker -> GET /tasks/:id`
- `LLM_BASE_URL` 必须带 `/v1`
- `rejected -> failed` 生命周期已确认

这些如果不通过记忆恢复，而要重新证明，成本很高：

- 需要重跑测试
- 需要重读 specs
- 需要重看代码
- 需要再写一次总结

### ROI 判断
**极高 ROI**。

这是“已经有 durable artifact / summary 的工程系统”最核心的收益点。

---

## 4. 在合回主仓阶段，记忆搜索极大减少了搬运错误

后半段的高价值记忆使用包括：

- 知道哪些文件属于 worktree，哪些属于主仓 docs
- 知道要分 4 个 commit，而不是一次大提交
- 知道 `05-learn` 文档不在 worktree 仓库里
- 知道先看 `git status / diff --stat / diff --name-only`

这使得“代码整理由实现问题”切换到“仓库操作问题”时，仍然保持连续性。

### ROI 判断
**高 ROI**。

因为这里避免的是：
- 错提交流程
- 提交边界混乱
- 把 docs 和代码仓混在一起
- 主仓脏工作区误覆盖

## 四、本 session 中低 ROI 的记忆搜索模式

## 1. 一次性大批量 `read-files` 预加载，ROI 偏低

从 summary 中可以看出，在某些阶段加载了大量文件，包括：

- 多个 skill 文档
- 多个 Pi README / docs / examples
- 多个项目 specs
- 多个 repo 文件
- 多个历史 solution / plan / brainstorm

问题不在于这些文件不重要，而在于：

- 它们并不是同一时刻都对当前决策有用
- 大量预加载会让活跃上下文里混入许多“冷信息”
- 很多文件后来并未直接影响当前这一轮修复

### ROI 判断
**中低 ROI**。

它的价值主要是建立广域背景，但对“继续修 runtime-session.test.ts”这种局部问题来说，成本偏高。

### 更优策略
应改为三层：

- **热上下文**：当前失败涉及的 3–6 个文件
- **温上下文**：必要时再调入相关 plan / solution / spec
- **冷上下文**：仅保留路径引用，不读全文

---

## 2. 技能规则 / 外部参考的全量确认，在局部 debug 场景容易过量

例如：
- 读取多个 skill 说明
- 再读取 Pi 文档与 examples
- 再读取其他 repo 文档

这些在“路线选择”阶段很值，但在“已经确定问题位于某个测试文件”之后，边际收益明显下降。

### ROI 判断
**中等偏低**。

不是无效，而是加载时机过早、过宽。

---

## 3. 已经确认方向后，重复解释历史决策，token 回报下降

长会话里一个常见问题是：
- 为了确保不丢上下文，反复把同一批背景重新讲一遍

本 session 里虽然 summary 已经做得很好，但后续若继续将大量旧背景重新嵌入回答，会出现：

- token 被历史决策占用
- 当前问题的“最小必要信息密度”下降
- 排错速度受影响

### ROI 判断
**低 ROI 候选**。

工程上应尽量把“已定事实”缩成：
- 结论
- 证据路径
- 不再重复的原因

而不是每轮重新叙述完整背景。

## 六、为什么这次 session 的记忆搜索总体值得

如果没有这些记忆恢复与工件检索，这次会话很可能退化成：

1. 重新读 docs
2. 重新读 specs
3. 重新判断 Pi 路线
4. 重新理解 worktree 改了什么
5. 重新跑 integration 证明
6. 重新确认 commit worker auto-review 逻辑
7. 重新分组整理文件

这会产生大量重复 token 成本，而且最致命的是：

- 容易把“已完成事实”误当成“待验证事项”
- 容易重复开无效调试分支
- 容易因为忘记 earlier constraints 而提出错误建议

所以从工程效率上看，本次 session 的记忆搜索是必要的，不是可选项。

## 七、本 session 暴露出的优化机会

## 1. 需要“记忆搜索预算”机制

建议把记忆读取分成预算桶：

- **摘要恢复预算**：优先使用，允许较高成本
- **局部 debug 预算**：严格限制，只读当前失败最相关文件
- **文档校验预算**：仅在路线不确定时开启
- **沉淀预算**：收尾阶段再读 solution / report 模板

这样可避免“刚进入局部修复，就把全套文档又读一遍”。

---

## 2. 需要“先路径，后全文”的 solution / plan 检索

这次 `05-learn` 已经用了类似思路，但还可以更彻底：

1. 先 grep 路径
2. 再看 frontmatter / 前 15 行
3. 最后只读 top 1–3 个

不要一上来就把多份全文读进上下文。

---

## 3. 需要把“已验证事实”做成更短的 hot memory

本 session 很多高价值事实其实可以压缩成一页：

- 当前 canonical upstream
- 当前最小闭环状态
- 当前测试证据
- 当前未解 blocker
- 当前主仓 / worktree 差异

这类 hot memory 最好控制在 **500–1500 tokens** 范围。

---

## 4. 需要把“调试历史”与“产品历史”分层

这次 summary 很强，但仍混合了：

- 产品路径决策
- 测试稳定化细节
- 合仓过程
- 每轮失败信息

更优结构应分成：

- **Product memory**：为什么这样设计
- **Execution memory**：已经做了哪些实现
- **Debug memory**：当前 blocker、最近失败、最小复现命令
- **Release memory**：commit、patch、cherry-pick、验证状态

这样下轮继续时，只读对应层即可。

## 八、对 Super Pi 的直接启发

这次 session 很适合反向验证 `super-pi` 当前 token/context 优化方向：

### 已被本 session 证明正确的方向
1. **压缩 summary / handoff-lite 非常值**
2. **工件化 memory 比会话内硬记忆更稳**
3. **grep-first / path-first 检索是高 ROI 的**
4. **跨阶段时，热/温/冷上下文分层是必要的**
5. **低 token continuation 只有在 durable artifacts 足够强时才成立**

### 本 session 进一步证明需要增强的能力
1. **记忆检索预算控制**
2. **按问题类型切换检索策略**
   - 路线选择：可读更多 docs
   - 局部 debug：严格收缩
   - 合仓收尾：偏 git / diff / status
3. **自动生成 debug-lite handoff**
   - 当前错误
   - 最小复现命令
   - 已排除假设
   - 最可能文件
4. **热记忆去重**
   - 避免每轮重复讲完整历史

我的判断是：

## 十、建议的后续动作

建议在 `super-pi` 后续设计里补 4 个能力：

1. **memory search budget**
   - 每轮限定读取量

2. **debug-lite handoff**
   - 专门给局部修复续接使用

3. **artifact tiering**
   - summary / handoff / report / solution 按热温冷检索

4. **evidence-first continuation**
   - 优先恢复：结论、证据、最小复现命令、当前 blocker
   - 延迟恢复：大段背景、长文档全文、弱相关历史

如果落实这 4 点，类似这次的长会话，Token ROI 还可以继续显著提高。
