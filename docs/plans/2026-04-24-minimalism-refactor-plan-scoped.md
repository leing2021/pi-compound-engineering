# 极简化重构计划（缩范围版）

> 日期：2026-04-24  
> 基于对 `docs/plans/2026-04-24-minimalism-refactor-plan.md` 的重新评估而收缩。  
> 目标：只解决**用户默认心智过重、入口重复、README 过长**这 3 个最直接问题，不在本轮同时推进状态模型重构与底层 token 优化。

---

## 一、这轮为什么要改

当前问题不是“功能不够”，而是“默认暴露面过大”：

1. **主入口有重复感**
   - `06-next` 与 `08-status` 都在回答“现在该干嘛”
   - 新用户不容易判断默认入口

2. **高级能力过早暴露**
   - `07-worktree` 很有用，但不该占据主心智
   - 默认流程不应要求用户先理解 git worktree

3. **README / README_CN 过长且重复**
   - 主路径、工具、辅助 skill、历史演进混在一起
   - 首次阅读成本偏高

4. **体验问题优先于内部优化**
   - 当前更需要先收短默认路径
   - 状态模型收口、schema description 压缩等内部优化延后评估

---

## 二、本轮只做什么

本轮只做 4 件事：

1. **收口状态入口**：让 `08-status` 成为默认的“状态 + 下一步”入口
2. **弱化高级能力**：将 `07-worktree` 下沉为 advanced / optional
3. **下沉解释型能力**：将 `09-help` 从默认心智中移出
4. **瘦身 README / README_CN**：把首页收成短入口，而不是总册

---

## 三、本轮明确不做什么

以下内容**不纳入本轮**：

1. 不做状态模型统一重构
   - 暂不统一 `workflow_state` / `session_checkpoint` / `context_handoff` / `session_history`

2. 不做全量 tool schema description 压缩
   - `extensions/ce-core/index.ts` 的 token 优化单独立项

3. 不做大规模工具分层治理
   - 可在 README 中轻量弱化 optional tools
   - 但不在本轮展开成架构级整理

4. 不做历史文档全面同步清洗
   - 只修改主入口相关文档
   - 旧报告 / 旧分析文档先保留

---

## 四、核心判断原则

### 1. 默认路径必须最短
凡是新用户第一次使用不必理解的内容，都不应出现在主叙事前排。

### 2. 优先消除重复入口
如果两个 skill 回答的是同一类问题，应优先合并，而不是并列保留。

### 3. 先改用户看到的，再改内部抽象
先验证默认体验是否显著变简单，再决定是否动底层结构。

### 4. 只做真减法，不做重新包装后的扩张
本轮禁止新增新概念、新入口、新包装层。

---

## 五、缩范围后的执行项

## Unit 01：收口 `06-next` → `08-status`

### 目标
让 `08-status` 成为唯一默认状态/下一步入口。

### 修改文件
- `skills/06-next/SKILL.md`
- `skills/08-status/SKILL.md`
- `README.md`
- `README_CN.md`

### 动作
- 将 `06-next` 的“推荐下一步”职责并入 `08-status`
- README/README_CN 不再把 `06-next` 作为主路径 skill 强调
- 如需保留 `06-next`，仅作为兼容入口，不再占据主叙事

### 完成标准
- 用户看到 `08-status` 就能理解：它既能看当前状态，也能告诉我下一步做什么
- README 主路径中不再并列强调 `06-next`

---

## Unit 02：弱化 `07-worktree`

### 目标
将 worktree 定位为高级工程能力，而不是默认流程心智。

### 修改文件
- `skills/07-worktree/SKILL.md`
- `README.md`
- `README_CN.md`

### 动作
- 将 `07-worktree` 标记为 advanced / optional
- 从 README 首页主叙事中后移
- 保留能力，但不作为新用户必须理解内容

### 完成标准
- 首页主流程不依赖 `07-worktree`
- 用户可以不用理解 worktree 也能理解 super-pi 主工作流

---

## Unit 03：下沉 `09-help`

### 目标
把帮助解释型 skill 从默认暴露面移到文档层。

### 修改文件
- `skills/09-help/SKILL.md`
- `README.md`
- `README_CN.md`

### 动作
- 将 `09-help` 从默认主路径移除
- 把必要解释迁移到 README 的简短“如何开始”部分
- 保留文档解释能力，但不让它成为主入口之一

### 完成标准
- 新用户不依赖 `09-help` 也能理解怎么开始

---

## Unit 04：README / README_CN 瘦身

### 目标
把 README 从“全量说明书”收成“短入口 + 进一步阅读链接”。

### 修改文件
- `README.md`
- `README_CN.md`

### 首页只保留
1. super-pi 是什么
2. 默认 5 个主 skill
3. 最短使用路径
4. `08-status` 作为状态/下一步统一入口
5. 高级能力入口（简短一行即可）
6. 指向详细 docs 的链接

### 删除或后移
- 过长的 tools 全表
- 非主路径 skill 的大篇幅说明
- 历史演进和版本说明型长内容
- 重复解释主流程的段落

### 完成标准
- README 首页明显变短
- 新用户能在更短篇幅内理解“我先用什么、再用什么”

---

## 六、推荐执行顺序

1. 先改 `08-status` / `06-next` 叙事收口
2. 再弱化 `07-worktree`
3. 再下沉 `09-help`
4. 最后统一瘦身 README / README_CN

原因：
- 先收入口，再改说明，避免 README 改完后又被 skill 定位变化反复打回

---

## 七、验收标准

本轮只看 5 个结果：

1. 用户默认只需要理解 5 个主 skill
2. `08-status` 成为唯一状态/下一步入口
3. `07-worktree` 不再干扰主心智
4. `09-help` 不再占据默认入口
5. README / README_CN 首页显著缩短，且主路径更清楚

---

## 八、本轮完成后再决定的事项

以下内容在本轮完成后单独复评：

1. 是否还有必要做状态模型收口
2. 是否需要压缩 tool schema descriptions
3. 是否要进一步做 Core tools / Optional tools 的正式分层文档
4. 是否要清理历史文档中的重复叙事

---

## 九、结论

这不是“全面极简化重构”，而是一次**面向用户默认心智的减法重构**。

先把用户真正会看到、会困惑、会犹豫的部分收短；
等第一轮效果成立，再决定是否继续做内部抽象与 token 成本优化。
