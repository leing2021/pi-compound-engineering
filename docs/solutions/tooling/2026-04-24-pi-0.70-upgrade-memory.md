# Problem

升级到 0.70.0 后，多个默认行为与兼容细节发生变化（例如终端进度条默认关闭、`noTools: "builtin"` 语义修正、GPT-5.5 Codex 能力与计费修正）。如果没有统一“升级记忆”，团队容易重复踩坑：

- 把“默认关闭”误判为功能失效
- 继续按旧语义理解 `--no-builtin-tools`
- 在扩展里继续持有过期 `ctx`，触发替换会话后的 stale context 错误
- 忽略 provider 兼容开关导致代理网关报错

# Context

本次 0.70.0 重点变化可归纳为 6 类：

1. 登录体验：`/login` provider 选择器支持模糊搜索，并新增 auth 来源标签。
2. 模型能力：新增 `openai-codex/gpt-5.5`，支持 `--thinking xhigh`，并修正优先层计费处理。
3. 默认行为：OSC 9;4 终端进度默认关闭，需在 `/settings` 打开 `terminal.showTerminalProgress=true`。
4. 工具选择语义：`--no-builtin-tools` / `createAgentSession({ noTools: "builtin" })` 仅禁用内建工具，不影响扩展工具。
5. 扩展生命周期：会话替换与退出顺序修复，强调扩展应使用 `withSession`，避免捕获旧 `ctx`。
6. 兼容与稳定性：增加多项 `models.json compat.*` 开关支持、Bedrock/Smithy HTTP/2 瞬时错误自动重试、若干 UI/渲染与安装路径修复。

# Solution

把 0.70.0 的变更固化为一份可复用升级清单，后续升级或排障时按此顺序核对：

## 1) 先核对“默认值变化”

- 若看不到终端进度：先检查 `terminal.showTerminalProgress` 是否显式开启。
- 不要再把“默认不显示进度”当成流式输出异常。

## 2) 再核对“语义修正”

- 需要禁用内建工具但保留扩展工具时，使用：
  - CLI: `--no-builtin-tools`
  - SDK: `createAgentSession({ noTools: "builtin" })`
- 若要禁用全部工具，才使用 `--no-tools`。

## 3) 模型与推理能力核对

- 选择 `openai-codex/gpt-5.5` 时，`--thinking xhigh` 不应再被降级到 `high`。
- 成本评估时考虑 GPT-5.5 Codex 的优先层价格修正（含倍率逻辑更新）。

## 4) 扩展代码核对（高频踩坑）

- 会话替换后，避免继续使用旧闭包里的 `pi/command ctx`。
- 需要跨替换执行后续逻辑时，统一改用 `withSession`。
- `/quit` 相关扩展 UI（footer、custom UI）需确保在上下文失效前完成清理。

## 5) Provider/代理兼容核对

针对 OpenAI/Anthropic 兼容代理，必要时使用：

- `compat.supportsLongCacheRetention`
- `compat.supportsEagerToolInputStreaming`
- `compat.sendSessionIdHeader: false`

用于处理网关不支持特定字段或 header 的情况。

## 6) 登录与配置引导核对

- 缺 auth、无 model 的提示链路应引导到 `/login`。
- Bedrock 用户按新版 `/login` 指引完成配置，不再只依赖 API Key 文案。

# Why this works

这份记忆把“功能新增”“默认值变化”“语义修正”“兼容开关”分层整理，能降低升级排查的分叉成本：

- 先排默认行为（最快）
- 再排参数语义（最容易误解）
- 最后排扩展生命周期和网关兼容（最隐蔽）

这样能把“升级后异常”快速定位到具体层面，而不是泛化为模型或网络问题。

# Prevention

- 每次大版本更新，固定产出一份“升级记忆”文档，至少覆盖：默认值、破坏性变更、语义修正、兼容开关。
- 扩展评审时加入一条硬规则：禁止在可跨会话生命周期逻辑中捕获旧 `ctx`。
- 对 `/login`、`/model`、`/quit` 相关流程保留最小回归清单，避免 UI 顺序调整后再次回归。
- 代理场景统一维护 `models.json compat` 参数基线，减少环境差异导致的偶发问题。
